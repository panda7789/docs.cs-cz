# <a name="appendix-a---transactions"></a><span data-ttu-id="28349-103">Příloha A – transakce</span><span class="sxs-lookup"><span data-stu-id="28349-103">Appendix A - Transactions</span></span>

<span data-ttu-id="28349-104">Windows Communication Foundation (WCF) podporuje distribuované transakce, což umožňuje provádět atomické operace v rámci více služeb.</span><span class="sxs-lookup"><span data-stu-id="28349-104">Windows Communication Foundation (WCF) supports distributed transactions, allowing you to perform atomic operations across multiple services.</span></span> <span data-ttu-id="28349-105">Tato funkce je založená na [DTC (Distributed Transaction Coordinator) Microsoftu](https://docs.microsoft.com/previous-versions/windows/desktop/ms684146(v=vs.85)).</span><span class="sxs-lookup"><span data-stu-id="28349-105">This functionality is based on the [Microsoft Distributed Transaction Coordinator](https://docs.microsoft.com/previous-versions/windows/desktop/ms684146(v=vs.85)).</span></span>

<span data-ttu-id="28349-106">V novějších mikroslužbách na šířku není tento typ automatizovaného zpracování distribuované transakce možný.</span><span class="sxs-lookup"><span data-stu-id="28349-106">In the newer microservices landscape, this type of automated distributed transaction processing isn't possible.</span></span> <span data-ttu-id="28349-107">Existuje příliš mnoho různých technologií, včetně relačních databází, NoSQL úložišť dat a systémů zasílání zpráv.</span><span class="sxs-lookup"><span data-stu-id="28349-107">There are too many different technologies involved, including relational databases, NoSQL data stores, and messaging systems.</span></span> <span data-ttu-id="28349-108">Může být také kombinace operačních systémů, programovacích jazyků a platforem používaných v jednom prostředí.</span><span class="sxs-lookup"><span data-stu-id="28349-108">There might also be a mix of operating systems, programming languages, and frameworks in use in a single environment.</span></span>

<span data-ttu-id="28349-109">Distribuovaná transakce WCF je implementací toho, co se říká dvoufázové [potvrzení (2PC)](https://en.wikipedia.org/wiki/Two-phase_commit_protocol).</span><span class="sxs-lookup"><span data-stu-id="28349-109">WCF distributed transaction is an implementation of what is known as a [two-phase commit (2PC)](https://en.wikipedia.org/wiki/Two-phase_commit_protocol).</span></span> <span data-ttu-id="28349-110">Transakce 2PC můžete implementovat ručně prostřednictvím koordinace zpráv napříč službami, vytvořením otevřených transakcí v rámci každé služby a odesláním zpráv potvrzení nebo vrácení změn v závislosti na úspěšnosti nebo selhání.</span><span class="sxs-lookup"><span data-stu-id="28349-110">You can implement 2PC transactions manually by coordinating messages across services, creating open transactions within each service, and sending commit or rollback messages, depending upon success or failure.</span></span> <span data-ttu-id="28349-111">Složitost týkající se správy 2PC se ale může zvyšovat exponenciálně, jak se budou vyvíjet systémy.</span><span class="sxs-lookup"><span data-stu-id="28349-111">However, the complexity involved in managing 2PC can increase exponentially as systems evolve.</span></span> <span data-ttu-id="28349-112">Otevřené transakce uchovávají zámky databáze, které můžou negativně ovlivnit výkon, nebo horší, což způsobí zablokování mezi službami.</span><span class="sxs-lookup"><span data-stu-id="28349-112">Open transactions hold database locks that can negatively affect performance, or, worse, cause cross-service deadlocks.</span></span>

<span data-ttu-id="28349-113">Pokud je to možné, je nejlepší vyhnout se distribuovaným transakcím úplně.</span><span class="sxs-lookup"><span data-stu-id="28349-113">If possible, it's best to avoid distributed transactions altogether.</span></span> <span data-ttu-id="28349-114">Pokud jsou tak propojeny dvě položky dat, jako by vyžadovaly atomické aktualizace, zvažte jejich zpracování pomocí stejné služby.</span><span class="sxs-lookup"><span data-stu-id="28349-114">If two items of data are so linked as to require atomic updates, consider handling them both with the same service.</span></span> <span data-ttu-id="28349-115">Tyto atomické změny použijte při použití jediné žádosti nebo zprávy do této služby.</span><span class="sxs-lookup"><span data-stu-id="28349-115">Apply those atomic changes by using a single request or message to that service.</span></span>

<span data-ttu-id="28349-116">Pokud to není možné, pak je jednou alternativou použití [vzoru Saga](https://microservices.io/patterns/data/saga.html).</span><span class="sxs-lookup"><span data-stu-id="28349-116">If that isn't possible, then one alternative is to use the [Saga pattern](https://microservices.io/patterns/data/saga.html).</span></span> <span data-ttu-id="28349-117">Ve Saga se aktualizace zpracovávají postupně. Po úspěšném dokončení každé aktualizace se aktivuje další.</span><span class="sxs-lookup"><span data-stu-id="28349-117">In a saga, updates are processed sequentially; as each update succeeds, the next one is triggered.</span></span> <span data-ttu-id="28349-118">Tyto aktivační události se dají rozšířit ze služby na službu nebo spravovat koordinátor Saga nebo Orchestrator.</span><span class="sxs-lookup"><span data-stu-id="28349-118">These triggers can be propagated from service to service, or managed by a saga coordinator or orchestrator.</span></span> <span data-ttu-id="28349-119">Pokud se aktualizace v jakémkoli okamžiku v rámci procesu nezdařila, služby, které již dokončí jejich aktualizace, uplatní konkrétní logiku pro jejich vrácení.</span><span class="sxs-lookup"><span data-stu-id="28349-119">If an update fails at any point during the process, the services that have already completed their updates apply specific logic to reverse them.</span></span>

<span data-ttu-id="28349-120">Další možností je použití návrhu řízeného doménou (DDD) a oddělení zodpovědnosti příkazů nebo dotazů (CQRS), jak je popsáno v [elektronické knize .NET mikroslužeb](https://docs.microsoft.com/dotnet/architecture/microservices/microservice-ddd-cqrs-patterns/).</span><span class="sxs-lookup"><span data-stu-id="28349-120">Another option is to use Domain Driven Design (DDD) and Command/Query Responsibility Segregation (CQRS), as described in the [.NET Microservices e-book](https://docs.microsoft.com/dotnet/architecture/microservices/microservice-ddd-cqrs-patterns/).</span></span> <span data-ttu-id="28349-121">Konkrétně použití doménových událostí nebo [zdrojů událostí](https://martinfowler.com/eaaDev/EventSourcing.html) může pomoci zajistit, že aktualizace budou konzistentně, pokud nejsou okamžitě použity.</span><span class="sxs-lookup"><span data-stu-id="28349-121">In particular, using domain events or [event sourcing](https://martinfowler.com/eaaDev/EventSourcing.html) can help to ensure that updates are consistently, if not immediately, applied.</span></span>
