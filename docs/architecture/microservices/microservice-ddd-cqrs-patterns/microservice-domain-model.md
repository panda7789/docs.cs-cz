## <a name="the-domain-entity-pattern"></a><span data-ttu-id="b808d-108">Vzor entity domény</span><span class="sxs-lookup"><span data-stu-id="b808d-108">The Domain Entity pattern</span></span>

<span data-ttu-id="b808d-109">Entity představují objekty domény a jsou primárně definovány jejich identitou, kontinuitou a stálostí v průběhu času, a nikoli pouze pomocí atributů, které je tvoří.</span><span class="sxs-lookup"><span data-stu-id="b808d-109">Entities represent domain objects and are primarily defined by their identity, continuity, and persistence over time, and not only by the attributes that comprise them.</span></span> <span data-ttu-id="b808d-110">Eric Evans říká, že "objekt primárně definovaný jeho identitou se nazývá entita."</span><span class="sxs-lookup"><span data-stu-id="b808d-110">As Eric Evans says, “an object primarily defined by its identity is called an Entity.”</span></span> <span data-ttu-id="b808d-111">Entity jsou v modelu domény velmi důležité, protože jsou základem pro model.</span><span class="sxs-lookup"><span data-stu-id="b808d-111">Entities are very important in the domain model, since they are the base for a model.</span></span> <span data-ttu-id="b808d-112">Proto byste je měli pečlivě identifikovat a navrhovat.</span><span class="sxs-lookup"><span data-stu-id="b808d-112">Therefore, you should identify and design them carefully.</span></span>

<span data-ttu-id="b808d-113">*Identita entity může protínat více mikroslužeb nebo ohraničených kontextů.*</span><span class="sxs-lookup"><span data-stu-id="b808d-113">*An entity’s identity can cross multiple microservices or Bounded Contexts.*</span></span>

<span data-ttu-id="b808d-114">Stejná identita (tj. stejná hodnota `Id`, i když možná není stejná doménová entita), může být modelována v rámci několika ohraničených kontextů nebo mikroslužeb.</span><span class="sxs-lookup"><span data-stu-id="b808d-114">The same identity (that is, the same `Id` value, although perhaps not the same domain entity) can be modeled across multiple Bounded Contexts or microservices.</span></span> <span data-ttu-id="b808d-115">To však neznamená, že stejná entita se stejnými atributy a logikou by byla implementována ve více ohraničených kontextech.</span><span class="sxs-lookup"><span data-stu-id="b808d-115">However, that does not imply that the same entity, with the same attributes and logic would be implemented in multiple Bounded Contexts.</span></span> <span data-ttu-id="b808d-116">Místo toho entity v každém ohraničeném kontextu omezují jejich atributy a chování na ty, které jsou požadovány v doméně daného vázaného kontextu.</span><span class="sxs-lookup"><span data-stu-id="b808d-116">Instead, entities in each Bounded Context limit their attributes and behaviors to those required in that Bounded Context’s domain.</span></span>

<span data-ttu-id="b808d-133">V jeho příspěvku [AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html)Novák Fowlera popisuje model domény anemic tímto způsobem:</span><span class="sxs-lookup"><span data-stu-id="b808d-133">In his post [AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html), Martin Fowler describes an anemic domain model this way:</span></span>

### <a name="the-aggregate-pattern"></a><span data-ttu-id="b808d-179">Agregovaný vzor</span><span class="sxs-lookup"><span data-stu-id="b808d-179">The Aggregate pattern</span></span>

<span data-ttu-id="b808d-180">Doménový model obsahuje clustery různých datových entit a procesů, které mohou řídit významnou oblast funkčnosti, jako je například plnění objednávky nebo inventarizace.</span><span class="sxs-lookup"><span data-stu-id="b808d-180">A domain model contains clusters of different data entities and processes that can control a significant area of functionality, such as order fulfillment or inventory.</span></span> <span data-ttu-id="b808d-181">Podrobná jednotka DDD je agregovaná, která popisuje cluster nebo skupinu entit a chování, které lze považovat za soudržnou jednotku.</span><span class="sxs-lookup"><span data-stu-id="b808d-181">A more fine-grained DDD unit is the aggregate, which describes a cluster or group of entities and behaviors that can be treated as a cohesive unit.</span></span>

<span data-ttu-id="b808d-182">Agregaci obvykle definujete v závislosti na transakcích, které potřebujete.</span><span class="sxs-lookup"><span data-stu-id="b808d-182">You usually define an aggregate based on the transactions that you need.</span></span> <span data-ttu-id="b808d-183">Klasický příklad je objednávka, která obsahuje také seznam položek objednávek.</span><span class="sxs-lookup"><span data-stu-id="b808d-183">A classic example is an order that also contains a list of order items.</span></span> <span data-ttu-id="b808d-184">Položka objednávky obvykle bude entitou.</span><span class="sxs-lookup"><span data-stu-id="b808d-184">An order item will usually be an entity.</span></span> <span data-ttu-id="b808d-185">Ale bude podřízenou entitou v rámci agregace pořadí, která bude také obsahovat entitu Order jako její kořenovou entitu, která se obvykle označuje jako agregovaná kořenová.</span><span class="sxs-lookup"><span data-stu-id="b808d-185">But it will be a child entity within the order aggregate, which will also contain the order entity as its root entity, typically called an aggregate root.</span></span>

<span data-ttu-id="b808d-186">Identifikace agregací může být pevná.</span><span class="sxs-lookup"><span data-stu-id="b808d-186">Identifying aggregates can be hard.</span></span> <span data-ttu-id="b808d-187">Agregace je skupina objektů, které musí být konzistentní společně, ale nelze vybrat pouze skupinu objektů a označit ji jako agregaci.</span><span class="sxs-lookup"><span data-stu-id="b808d-187">An aggregate is a group of objects that must be consistent together, but you cannot just pick a group of objects and label them an aggregate.</span></span> <span data-ttu-id="b808d-188">Musíte začít s konceptem domény a zamyslete se nad entitami, které se používají v nejběžnějších transakcích souvisejících s tímto konceptem.</span><span class="sxs-lookup"><span data-stu-id="b808d-188">You must start with a domain concept and think about the entities that are used in the most common transactions related to that concept.</span></span> <span data-ttu-id="b808d-189">Ty entity, které je potřeba vyhodnotit jako nevhodné, jsou, které tvoří agregaci.</span><span class="sxs-lookup"><span data-stu-id="b808d-189">Those entities that need to be transactionally consistent are what forms an aggregate.</span></span> <span data-ttu-id="b808d-190">Nejlepším způsobem, jak identifikovat agregace, je pravděpodobně myslím na operace s transakcemi.</span><span class="sxs-lookup"><span data-stu-id="b808d-190">Thinking about transaction operations is probably the best way to identify aggregates.</span></span>

### <a name="the-aggregate-root-or-root-entity-pattern"></a><span data-ttu-id="b808d-191">Model agregované kořenové nebo kořenové entity</span><span class="sxs-lookup"><span data-stu-id="b808d-191">The Aggregate Root or Root Entity pattern</span></span>

<span data-ttu-id="b808d-192">Agregace se skládá z aspoň jedné entity: agregační kořen, označovaný také jako kořenová entita nebo primární entita.</span><span class="sxs-lookup"><span data-stu-id="b808d-192">An aggregate is composed of at least one entity: the aggregate root, also called root entity or primary entity.</span></span> <span data-ttu-id="b808d-193">Kromě toho může mít více podřízených entit a objektů hodnot se všemi entitami a objekty, které pracují společně pro implementaci požadovaných chování a transakcí.</span><span class="sxs-lookup"><span data-stu-id="b808d-193">Additionally, it can have multiple child entities and value objects, with all entities and objects working together to implement required behavior and transactions.</span></span>

<span data-ttu-id="b808d-194">Účelem agregovaného kořene je zajistit konzistenci agregace; měl by být jediným vstupním bodem pro aktualizace agregace prostřednictvím metod nebo operací v agregační kořenové třídě.</span><span class="sxs-lookup"><span data-stu-id="b808d-194">The purpose of an aggregate root is to ensure the consistency of the aggregate; it should be the only entry point for updates to the aggregate through methods or operations in the aggregate root class.</span></span> <span data-ttu-id="b808d-195">Změny entit v rámci agregace byste měli provádět pouze prostřednictvím agregovaného kořene.</span><span class="sxs-lookup"><span data-stu-id="b808d-195">You should make changes to entities within the aggregate only via the aggregate root.</span></span> <span data-ttu-id="b808d-196">Je to modul pro sjednocení konzistence agregace a zvažuje všechna pravidla invariantní a konzistence, která může být potřeba splnit v agregaci.</span><span class="sxs-lookup"><span data-stu-id="b808d-196">It is the aggregate’s consistency guardian, considering all the invariants and consistency rules you might need to comply with in your aggregate.</span></span> <span data-ttu-id="b808d-197">Pokud změníte podřízenou entitu nebo objekt hodnoty nezávisle, agregovaný kořen nemůže zajistit, aby agregace byla v platném stavu.</span><span class="sxs-lookup"><span data-stu-id="b808d-197">If you change a child entity or value object independently, the aggregate root cannot ensure that the aggregate is in a valid state.</span></span> <span data-ttu-id="b808d-198">Vypadá to, že tabulka s volnou nožkou.</span><span class="sxs-lookup"><span data-stu-id="b808d-198">It would be like a table with a loose leg.</span></span> <span data-ttu-id="b808d-199">Zachování konzistence je hlavním účelem agregovaného kořene.</span><span class="sxs-lookup"><span data-stu-id="b808d-199">Maintaining consistency is the main purpose of the aggregate root.</span></span>
