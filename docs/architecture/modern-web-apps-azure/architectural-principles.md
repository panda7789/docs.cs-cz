<span data-ttu-id="c6405-171">**Neznalost přetrvávání** (PI) odkazuje na typy, které je třeba zachovat, ale jejichž kód není ovlivněn volbou technologie trvalosti.</span><span class="sxs-lookup"><span data-stu-id="c6405-171">**Persistence ignorance** (PI) refers to types that need to be persisted, but whose code is unaffected by the choice of persistence technology.</span></span> <span data-ttu-id="c6405-172">Tyto typy v rozhraní .NET jsou někdy označovány jako plain old CLR objects (POCO), protože není nutné dědit z určité základní třídy nebo implementovat určité rozhraní.</span><span class="sxs-lookup"><span data-stu-id="c6405-172">Such types in .NET are sometimes referred to as Plain Old CLR Objects (POCOs), because they do not need to inherit from a particular base class or implement a particular interface.</span></span> <span data-ttu-id="c6405-173">Neznalost trvalosti je cenná, protože umožňuje zachovat stejný obchodní model několika způsoby a nabízí další flexibilitu aplikace.</span><span class="sxs-lookup"><span data-stu-id="c6405-173">Persistence ignorance is valuable because it allows the same business model to be persisted in multiple ways, offering additional flexibility to the application.</span></span> <span data-ttu-id="c6405-174">Možnosti trvalosti se mohou v průběhu času měnit, z jedné databázové technologie na jinou, nebo mohou být kromě toho, s čím aplikace byla spuštěna, vyžadovány další formy trvalosti (například použití mezipaměti Redis nebo Azure Cosmos DB kromě relační databáze).</span><span class="sxs-lookup"><span data-stu-id="c6405-174">Persistence choices might change over time, from one database technology to another, or additional forms of persistence might be required in addition to whatever the application started with (for example, using a Redis cache or Azure Cosmos DB in addition to a relational database).</span></span>

<span data-ttu-id="c6405-175">Některé příklady porušení tohoto principu zahrnují:</span><span class="sxs-lookup"><span data-stu-id="c6405-175">Some examples of violations of this principle include:</span></span>

- <span data-ttu-id="c6405-180">Vlastnosti vyžadující virtuální klíčové slovo.</span><span class="sxs-lookup"><span data-stu-id="c6405-180">Properties requiring virtual keyword.</span></span>

- <span data-ttu-id="c6405-181">Požadované atributy specifické pro trvalost.</span><span class="sxs-lookup"><span data-stu-id="c6405-181">Persistence-specific required attributes.</span></span>

<span data-ttu-id="c6405-182">Požadavek, aby třídy mají některé z výše uvedených funkcí nebo chování přidává spojení mezi typy, které mají být trvalé a výběr technologie trvalosti, takže je obtížnější přijmout nové strategie přístupu k datům v budoucnu.</span><span class="sxs-lookup"><span data-stu-id="c6405-182">The requirement that classes have any of the above features or behaviors adds coupling between the types to be persisted and the choice of persistence technology, making it more difficult to adopt new data access strategies in the future.</span></span>

### <a name="bounded-contexts"></a><span data-ttu-id="c6405-183">Ohraničené kontexty</span><span class="sxs-lookup"><span data-stu-id="c6405-183">Bounded contexts</span></span>

<span data-ttu-id="c6405-184">**Ohraničené kontexty** jsou centrální vzor v návrhu řízeném doménou.</span><span class="sxs-lookup"><span data-stu-id="c6405-184">**Bounded contexts** are a central pattern in Domain-Driven Design.</span></span> <span data-ttu-id="c6405-185">Poskytují způsob řešení složitosti ve velkých aplikacích nebo organizacích rozdělením do samostatných koncepčních modulů.</span><span class="sxs-lookup"><span data-stu-id="c6405-185">They provide a way of tackling complexity in large applications or organizations by breaking it up into separate conceptual modules.</span></span> <span data-ttu-id="c6405-186">Každý koncepční modul pak představuje kontext, který je oddělen od jiných kontextů (proto ohraničené) a může vyvíjet nezávisle.</span><span class="sxs-lookup"><span data-stu-id="c6405-186">Each conceptual module then represents a context that is separated from other contexts (hence, bounded), and can evolve independently.</span></span> <span data-ttu-id="c6405-187">Každý ohraničený kontext by měl mít v ideálním případě možnost zvolit své vlastní názvy pro koncepty v něm a měl by mít výhradní přístup k vlastnímu úložišti trvalosti.</span><span class="sxs-lookup"><span data-stu-id="c6405-187">Each bounded context should ideally be free to choose its own names for concepts within it, and should have exclusive access to its own persistence store.</span></span>

<span data-ttu-id="c6405-188">Minimálně jednotlivé webové aplikace by se měly snažit být jejich vlastní ohraničený kontext, s vlastním úložištěm trvalosti pro jejich obchodní model, spíše než sdílet databázi s jinými aplikacemi.</span><span class="sxs-lookup"><span data-stu-id="c6405-188">At a minimum, individual web applications should strive to be their own bounded context, with their own persistence store for their business model, rather than sharing a database with other applications.</span></span> <span data-ttu-id="c6405-189">Komunikace mezi ohraničené kontexty probíhá prostřednictvím programových rozhraní, nikoli prostřednictvím sdílené databáze, která umožňuje obchodní logiku a události probíhat v reakci na změny, ke kterým dochází.</span><span class="sxs-lookup"><span data-stu-id="c6405-189">Communication between bounded contexts occurs through programmatic interfaces, rather than through a shared database, which allows for business logic and events to take place in response to changes that take place.</span></span> <span data-ttu-id="c6405-190">Ohraničené kontexty mapovat úzce mikroslužeb, které jsou také v ideálním případě implementovány jako jejich vlastní individuální ohraničené kontexty.</span><span class="sxs-lookup"><span data-stu-id="c6405-190">Bounded contexts map closely to microservices, which also are ideally implemented as their own individual bounded contexts.</span></span>
