# <a name="abstractions-abstract-types-and-interfaces"></a><span data-ttu-id="a8e12-102">Abstrakce (abstraktní typy a rozhraní)</span><span class="sxs-lookup"><span data-stu-id="a8e12-102">Abstractions (Abstract Types and Interfaces)</span></span>
<span data-ttu-id="a8e12-103">Abstrakce je typ, který popisuje kontrakt, ale neposkytuje úplnou implementaci kontraktu.</span><span class="sxs-lookup"><span data-stu-id="a8e12-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="a8e12-104">Abstrakce jsou obvykle implementovány jako abstraktní třídy nebo rozhraní a jsou dodávány s dobře definovanou sadou Referenční dokumentace popisující požadovanou sémantiku typů, které implementují kontrakt.</span><span class="sxs-lookup"><span data-stu-id="a8e12-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="a8e12-105">Mezi nejdůležitější abstrakce v .NET Framework patří <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>a <xref:System.Object>.</span><span class="sxs-lookup"><span data-stu-id="a8e12-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>

 <span data-ttu-id="a8e12-106">Rozhraní můžete roztáhnout implementací konkrétního typu, který podporuje kontrakt abstrakce, a použitím tohoto konkrétního typu s rozhraními API architektury, které spotřebovávají (pracuje na) abstrakce.</span><span class="sxs-lookup"><span data-stu-id="a8e12-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>

 <span data-ttu-id="a8e12-107">Smysluplná a užitečná abstrakce, která může vydržet test času, je velmi obtížné navrhnout.</span><span class="sxs-lookup"><span data-stu-id="a8e12-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="a8e12-108">Hlavní obtížnost je získání správné sady členů, ne více a méně.</span><span class="sxs-lookup"><span data-stu-id="a8e12-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="a8e12-109">Pokud má abstrakce příliš mnoho členů, je obtížné nebo dokonce možné ji implementovat.</span><span class="sxs-lookup"><span data-stu-id="a8e12-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="a8e12-110">Pokud má příliš málo členů pro přislíbenou funkci, je to v mnoha zajímavých scénářích zbytečné.</span><span class="sxs-lookup"><span data-stu-id="a8e12-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>

 <span data-ttu-id="a8e12-111">Příliš mnoho abstrakcí v rámci architektury má také negativní vliv na použitelnost rozhraní.</span><span class="sxs-lookup"><span data-stu-id="a8e12-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="a8e12-112">Je často poměrně obtížné pochopit abstrakci, aniž byste pochopili, jak se vejde na větší obrázek konkrétních implementací a rozhraní API, která na abstrakci pracují.</span><span class="sxs-lookup"><span data-stu-id="a8e12-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="a8e12-113">Názvy abstrakcí a jejich členů jsou také nutně abstraktní, což často usnadňuje jejich nepřístupné a nepřístupné, aniž by bylo nutné nejprve pochopit širší kontext jejich využití.</span><span class="sxs-lookup"><span data-stu-id="a8e12-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>

 <span data-ttu-id="a8e12-114">Abstrakce však poskytují extrémně výkonnou rozšiřitelnost, kterou jiné mechanismy rozšíření nemůžou často porovnat.</span><span class="sxs-lookup"><span data-stu-id="a8e12-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="a8e12-115">Jsou v jádru mnoha vzorů architektury, jako jsou moduly plug-in, inverze řídicích prvků (IoC), kanály atd.</span><span class="sxs-lookup"><span data-stu-id="a8e12-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="a8e12-116">Jsou také mimořádně důležité pro testování architektur.</span><span class="sxs-lookup"><span data-stu-id="a8e12-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="a8e12-117">Dobré abstrakce umožňují, aby byly pro účely testování částí využívány těžké závislosti.</span><span class="sxs-lookup"><span data-stu-id="a8e12-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="a8e12-118">V souhrnu jsou abstrakce zodpovědné za hledané – po bohatosti moderních rozhraní orientovaných na objekty.</span><span class="sxs-lookup"><span data-stu-id="a8e12-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>

 <span data-ttu-id="a8e12-119">❌ neposkytuje abstrakce, pokud nejsou testovány vývojem několika konkrétních implementací a rozhraní API, které spotřebovávají abstrakce.</span><span class="sxs-lookup"><span data-stu-id="a8e12-119">❌ DO NOT provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>

 <span data-ttu-id="a8e12-120">✔️ Při navrhování abstrakce zvolit pečlivě mezi abstraktní třídou a rozhraním.</span><span class="sxs-lookup"><span data-stu-id="a8e12-120">✔️ DO choose carefully between an abstract class and an interface when designing an abstraction.</span></span>

 <span data-ttu-id="a8e12-121">✔️ Zvažte poskytnutí referenčních testů pro konkrétní implementace abstrakcí.</span><span class="sxs-lookup"><span data-stu-id="a8e12-121">✔️ CONSIDER providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="a8e12-122">Tyto testy by uživatelům umožnily testovat, zda jejich implementace správně implementují kontrakt.</span><span class="sxs-lookup"><span data-stu-id="a8e12-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
